## 第4页：校园外卖市场与传统痛点

先看几个数字。2025年中国外卖市场规模已经突破四千五百亿，年增长率超过22%。高校群体的外卖渗透率超过70%——也就是说，十个大学生里有七个每周都在点外卖。一个头部高校日均外卖订单量超过一百二十万单。这是巨大的刚需市场，而且还在快速增长。

但是痛点也非常突出，我把它归纳为三个字：**进不来、乱、丢。**

先说**进不来**。每一所大学都有门禁制度，校外骑手到了校门口，保安不让进宿舍区。为什么？安全考虑——不能让校外人员随意进出学生宿舍。这就导致配送链在校门口断了，学生得自己从宿舍走到校门口去取，来回几百米，冬天下雪夏天暴晒，体验非常差。

第二是**乱**。中午十一点半下课以后，三十多个骑手同时聚在校门口，每个人手里好几单，餐往地上一放，学生围上来翻找自己的那份。人多手杂，场面完全失控。骑手急着送下一单，放下就走，根本不管是不是本人取走的。

第三是**丢**。外卖堆在地上没人管理，被拿错是家常便饭。有统计数据显示，校园外卖的日均丢失率在3%到5%之间。什么意思？每天有一百单外卖，就有三到五单被人错拿或者干脆丢了。你花十八块点的黄焖鸡，可能被隔壁不认识的同学顺手拎走了。

这三个痛点，表面看是管理问题，但本质上反映的是同一个结构缺陷：**校外配送链和校内宿舍楼之间，缺了一个规范的衔接环节。** 我们的系统就是来解决这个问题的。

---

## 第5页：解决方案——"两段式"配送模型

怎么解决？我们的思路是：把一条完整的配送链拆成**两段**，在中间加一个缓冲中转站。

我打个比方大家就懂了。你们肯定都用过快递柜吧？快递从外省运到你家小区，是快递公司的干线物流。到了小区之后，放进丰巢快递柜。然后你自己下楼从柜子里取出来，这就是最后一小段。我们做的就是把同样的模式搬到校园外卖里。

具体来说：

**第一段——干线运输。** 食堂档口的商家出餐以后，我们的干线骑手去取餐。他一次性可以拿好几单，骑着电动车送到宿舍楼下的智能寄存柜。他不需要进宿舍楼——把餐放进柜子，任务就完成了。全程在校外公共区域，不涉及宿舍安全管控的问题。

**第二段——楼栋配送。** 楼栋骑手从寄存柜里把包裹取出来，送到学生寝室门口。这个楼栋骑手本身就是学校勤工俭学的学生，他对这栋楼的布局了如指掌——哪一层、哪个房间、怎么走最快——轻车熟路。

中间那个寄存柜，就是两段之间的"缓冲层"。它起到了几个关键作用：第一，物理隔离了校外人员和宿舍区——干线骑手只到柜子为止；第二，解耦了两段配送——干线骑手不用等学生下来取，放下就走，楼栋骑手也不用跑去校门口，在楼下就能取货；第三，容量有限这个"劣势"反而被我们利用，变成了自动调节订单流量的机制——柜子满了就不能再下单，防止高峰期无限堆积。

总结这个设计的核心价值就一句话：**校外不进校，楼内专人送，全程可追溯。**

---

## 第6页：六态订单状态机

有了两段配送模型，订单该怎么流转？我们设计了一个**六态状态机**，把订单从创建到完成的全生命周期管起来。

我按时间顺序说一遍，大家把它想象成一个快递包裹的物流轨迹：

第一步，**Paid——已支付。** 你选好了菜、付了钱，订单在系统里生成了。这个时候还没有骑手接单，订单处于待分配状态。

第二步，**Stage1_Assigned——干线配送中。** 系统分配了一个干线骑手，他正在去商家档口取餐。这个阶段大家关心的是："我的餐出食堂了吗？"

第三步，**Arrived_At_Point——已入柜。** 干线骑手把餐送到了宿舍楼下的寄存柜，放进去、关上门。系统记录下入柜的准确时间。这个状态很重要——它标志着干线段的任务完成，干线骑手可以被释放去接下一单了。

第四步，**Stage2_Assigned——楼栋配送中。** 楼栋骑手从柜子里取出了包裹，正在往你宿舍走。这个阶段通常只需要几分钟，因为楼栋骑手就在这栋楼附近。

第五步，**Completed——已完成。** 餐送到了，你拿到了，订单完结。系统记录下签收时间。

第六步，**Cancelled——已取消。** 在任何非终态节点，你都可以取消订单。系统会自动退款、恢复商家库存、释放占用的寄存点容量、释放所有关联骑手——这些操作都是存储过程自动完成的。

六个状态、三个时间戳——创建时间、入柜时间、签收时间。全链路每一段的责任人是谁、花了多长时间，清清楚楚，随时可查。这就是状态机的价值：**把业务流程变成可审计的数据。**

## 第8页：系统三层架构

我们采用了经典的三层架构，每一层的职责非常清晰。

最上面是**展现层**。我们做了一个基于 Flask 和 ECharts 的 Web 管理大屏，五个核心 KPI、订单状态分布、寄存点饱和度监控、商家排行榜——所有数据30秒自动刷新。你不需要安装任何软件，浏览器打开就能看，手机也能看。

中间是**业务逻辑层**。Python Flask 框架搭建了 RESTful API，负责处理下单、派单、状态流转这些核心业务。这一层还集成了 DeepSeek 大模型，实现自然语言查询数据——你说"今天卖了多少钱"，系统自动转成 SQL 去查数据库。

最下面是**数据层**。MySQL 8.0 InnoDB 引擎，这是我们整个系统最核心、也是花了最多心血的一层。七张业务表、四个 B-Tree 索引、七个触发器、四个存储过程、两个分析视图——这些不是随便堆上去的，每一项都是针对具体问题设计的。我接下来会逐一展开。

打个比喻帮助理解：展现层是你开车时看到的仪表盘，业务层是发动机和变速箱，数据层是底盘和安全气囊。真正保证你这趟行程不出事的，是底盘——数据层的约束和校验。

不管应用层代码怎么写、不管有没有bug，数据层都能保证最终写到磁盘上的数据是正确的。这是我们整个设计哲学的核心：**约束写在离数据最近的地方。**

---

## 第9页：技术栈选型

四块核心技术选型，每一块我简单说一下选择理由。

**Flask**：轻量、灵活、启动快。课程项目不需要 Spring Boot 那种企业级重型框架，Flask 的核心只有几个文件，一个 `app.py` 就撑起了整个后端。但它该有的扩展全都有——CORS 跨域、RESTful 路由、模板渲染。

**MySQL 8.0**：选它不是因为熟悉，而是因为我们需要的能力只有它给得了。窗口函数 `RANK() OVER`、公共表表达式 CTE、FOR UPDATE 行级锁——这三个是支撑我们系统核心功能的基础设施。低版本的 MySQL 不支持窗口函数，PostgreSQL 也行但我们的触发器需要用到 SIGNAL 语法。MySQL 8.0 刚好全部满足。

**ECharts**：Apache 开源项目，一张 CDN 链接就搞定，不用 npm install。折线图、柱状图、饼图、仪表盘——图表类型齐全，交互流畅，而且中文文档非常完善。

**DeepSeek**：国产大模型，API 调用成本极低，一次 Text-to-SQL 请求不到一分钱。我们用的是 DeepSeek-Chat 模型，响应速度快，SQL 生成质量稳定。

## 第10页：数据库E-R图

这是我们的数据库 E-R 图，七张核心表，严格遵循第三范式。

大家可以看到实体之间的映射关系非常清晰：学生（users）下订单（orders），订单包含若干明细项（order_items），每个明细项对应一道菜品（dishes），菜品从属于商家（merchants）。订单在转运环节暂存于寄存点（pickup_points），两段配送分别由干线骑手和楼栋骑手（riders）完成。

五个外键严格约束了实体间的引用关系——你不能插入一个指向不存在用户的订单，也不能把订单分配给一个不存在的寄存点。这些外键在数据库层面强制执行，不是靠应用层代码来检查的。

七张表、没有冗余字段、没有传递依赖、所有非主属性完全函数依赖于主键——这就是第三范式的标准要求。为什么我们这么强调范式？因为冗余数据是数据不一致的温床。一个学生改了手机号，如果存在两处，只改了一处——数据就脏了。

## 第12页：核心表结构：寄存点表（pickup_points）

先说最特别、也最能体现我们设计思路的一张表——**寄存点表**。

这个表的结构并不复杂：`point_id` 是主键，`point_name` 是寄存点名称比如"1期智能寄存柜"，`location` 是物理位置，`capacity` 是最大容量也就是这个柜子有多少个格子，`current_packages` 是当前里面有多少个包裹。

但是这条约束是整张表的灵魂：

```sql
CONSTRAINT chk_capacity CHECK (current_packages <= capacity)
```

什么意思呢？比如说1期寄存柜有80个格子，那 `capacity = 80`。这条 CHECK 约束保证：**current_packages 这个数字，在任何情况下、任何时刻、任何人来操作，都绝对不可能大于80。**

我再强调一遍这个"任何"。哪怕应用层代码有bug，哪怕有人绕过 Flask 直接拿 MySQL 客户端连上去写数据，哪怕并发场景下两个请求同时想往第80个格子里塞东西——只要数据要写入磁盘，数据库引擎就会在写入的瞬间执行这个 CHECK。80个格子，第81个包裹绝对写不进去。

传统的做法是什么？在 Python 代码里写一句 `if current < capacity`。代码可能写错、可能被绕过、可能在并发场景下判断失效。我们的做法是把约束写在离数据最近的地方——数据库引擎层。这是整个项目最重要的设计理念之一，后面会反复出现。

另外还有一个约束 `CHECK (current_packages >= 0)`——防止包裹数变成负数。双保险。

---

## 第13页：核心表结构：骑手表（riders）

骑手表的设计有两个关键决策。

第一个是 `rider_type` 字段。我们用 ENUM 枚举类型，硬编码了两个合法值：`Stage1_Trunk` 代表干线骑手，`Stage2_Floor` 代表楼栋骑手。它在数据库层面就限死了——你不可能插入一个类型为 `Stage3_Unknown` 的骑手，数据库直接拒绝。这个约束比任何应用层校验都可靠。

第二个是 `status` 字段。三个状态：Idle 表示空闲、可以接单；Delivering 表示配送中、正在跑单；Offline 表示离线、今天不上班。

这个 status 字段最妙的地方在哪呢？它**不是人工维护的**。调度员不需要在系统里点"释放骑手"按钮。订单状态一变，触发器自动把骑手状态改了——这个设计我马上在触发器章节展开讲。

全线干骑手8个、楼栋骑手7个，共15个骑手支撑整个校园的配送运力。

## 第14页：核心表结构：订单表（orders）

订单表是整个系统的"数据中枢"——所有业务流程最终都体现在这张表的一行记录上。

几个核心设计：

`order_status` 同样用了 ENUM，限死六种合法状态。为什么用 ENUM 而不是 VARCHAR？因为 VARCHAR 可以写入任意字符串——哪天代码出了bug，写入了一个 `"Completted"`（多了一个t），系统就找不到这条记录了。ENUM 从根本上杜绝了脏数据的可能性。

`stage1_rider_id` 和 `stage2_rider_id` 分别记录两段配送分别由哪个骑手负责。两个字段都是外键，指向同一张 riders 表——但通过触发器约束保证了一个必须是干线骑手、一个必须是楼栋骑手。

三个时间戳设计：`created_at` 下单时间、`stage1_completed_at` 干线送达时间、`stage2_completed_at` 最终签收时间。三个时间节点完整记录了一条订单的物流时间轴。如果哪天学生投诉"我等了一个小时还没收到"，我们拿出这三个时间戳，马上能定位是卡在干线段还是楼栋段。

你可以把订单表想象成快递单号——从下单到签收，每一个节点的责任人是谁、花了多少时间，一目了然。

---

## 第15页：核心表结构：学生用户表（users）

学生用户表的核心约束是 `CHECK (balance >= 0)`。

这个约束保证了什么？你的校园卡钱包余额永远不会变成负数。比如说你余额还剩5块钱，你同时下了两个订单——两个并发请求同时读到了余额5块，都觉得"够付"，然后同时扣款。如果不加约束，余额就变成负数了。

但有 `CHECK (balance >= 0)` 在，第二个扣款操作在写入磁盘时会被数据库引擎直接拒绝——因为写入后的余额会变成负数。这就是把约束写在数据库层的威力。

另外 `phone UNIQUE` 保证了手机号唯一——一个手机号只能注册一个账号。InnoDB 会自动为 UNIQUE 列创建 B-Tree 索引，手机号登录查询也能加速。

`dorm_building` 和 `room_number` 两个字段精准定位到寝室——"1期5栋 A302"。为什么需要这么精准？因为楼栋骑手要根据这个地址把餐送到门口。干线段看寄存点就够了，楼栋段必须精确到房间号。

---

## 第16页：核心表结构：商家表（merchants）

商家表有一个容易被忽略但很重要的约束：

```sql
CHECK (rating >= 1.0 AND rating <= 5.0)
```

这个约束保证了商家的评分永远在1.0到5.0之间。你永远不可能看到某个商家评分是9.9分——因为数据库根本不会接受这个写入。

有人可能会问：前端表单不是也有校验吗？确实有，但前端校验是防君子不防小人。HTTP 请求可以伪造，API 参数可以篡改。唯一你不可能绕过的是数据库——数据在写入磁盘的那一刻，数据库引擎执行 CHECK，不满足条件就拒绝。数据脏不脏，数据库说了算，不是前端表单说了算。

`address` 字段记录了商家的具体档口位置——"丁香餐厅一楼3号档口"。这个信息是为干线骑手准备的——他需要知道去哪取餐。

---

## 第17页：核心表结构：菜品表（dishes）

菜品表，也叫"menu表"，是**整个防超卖体系的核心战场**。

三个 CHECK 约束，一个都不能少：

`CHECK (price > 0)`：白送的东西不存在，价格必须大于零。

`CHECK (stock >= 0)`：这道菜还剩多少份——这个数字永远不会变成负数。这个约束是所有防超卖机制的最后一层兜底。

`CHECK (status IN (0, 1))`：0 表示这道菜今天下架了、不卖了，1 表示正常售卖。你在前端看到的菜单，其实就是 `WHERE status = 1` 的结果。

stock 这个字段，是整个系统并发控制的焦点。每次有人下单，数据库不是简单地 `UPDATE dishes SET stock = stock - 1`——在这之前，要先 `SELECT stock FROM dishes WHERE dish_id = ? FOR UPDATE`，加一把排他锁，锁住这一行，然后检查库存够不够。够了才扣，不够就拒绝。整个过程在一个事务里完成，是原子化的。

这个机制我马上在触发器章节详细展开。但这里先点出一个关键：FOR UPDATE 锁住的是 dishes 表的行，不是 orders 表的行——锁是加在库存数据上的，不是加在订单数据上的。这是精心选择的结果，后面会解释。

---

## 第18页：核心表结构：订单明细表（order_items）

订单明细表有一个非常经典的设计模式——**价格快照**。

`price_at_order` 这个字段，记录的是下单那一瞬间的菜品价格，而不是当前的菜品价格。

为什么需要这个设计？举个例子。你今天点了一份黄焖鸡，下单的时候价格是18块。第二天商家涨价了，黄焖鸡变成了22块。如果你回去看历史订单记录，它应该显示18块还是22块？当然是18块——你当时付的就是18块。

如果把订单金额关联到 dishes 表的价格字段，商家一改价，历史订单全部跟着变——这是严重的数据一致性问题。电商行业的标准做法就是"价格快照"：下单那一刻，把当前价格复制一份，存到订单明细表里。从此以后，这道菜涨跌跟你这份历史订单无关。

`ON DELETE CASCADE` 也是一个精心设计的细节。一个订单被删除了，它下面的所有明细行自动跟着删除——不需要应用层写代码逐条清理。数据库引擎替你维护引用完整性。

另外注意：`BEFORE INSERT ON order_items` 是库存防超卖触发器的挂载点。触发器不是挂在 orders 表上，而是挂在 order_items 表上——因为库存检查是在写入订单明细的时候触发的，不是在创建订单主记录的时候。这个时序设计很重要。

## 第19页：04 索引策略与性能优化（章节分隔页）

七张表讲完了，接下来进入第四部分——性能优化。数据库不是表建完了就完事了，查询慢了用户就会骂。索引策略是数据库设计中必须提前规划的一环。

## 第20页：4个B-Tree索引总览

我们没有盲目建一堆索引。索引不是越多越好——每多一个索引，INSERT 和 UPDATE 就要多维护一棵 B-Tree，写入性能会下降。我们的原则是：**只给最高频的查询路径建索引。**

第一个索引，`idx_orders_status`，建在 `order_status` 列上。为什么？因为我们的大屏最核心的查询就是"今天各种状态的订单各有多少"。没有索引的话，MySQL 要全表扫描几万行数据；有了索引，直接走 B-Tree 索引扫描，毫秒级别。

第二个索引，`idx_orders_created`，建在 `created_at` 列上。大屏支持切换时间范围——"今天"、"最近7天"、"最近30天"——每个范围都是对 `created_at` 的范围查询。B-Tree 索引对范围查询天然友好。

第三个索引，`idx_dishes_merchant`，是一个复合索引，建在 `(merchant_id, status)` 两个字段上。前端浏览菜单的时候会执行 `SELECT * FROM dishes WHERE merchant_id = ? AND status = 1`。复合索引遵从最左前缀原则——这个索引同时加速了"查某个商家的在售菜品"和"查某个商家的全部菜品"两种查询模式。

第四个索引，`idx_orders_point_status`，同样是复合索引，建在 `(pickup_point_id, order_status)` 上。这是我们根据大屏性能测试加上的一个关键索引。容量检查视图需要 JOIN orders 表和 pickup_points 表，按寄存点和订单状态聚合。没有这个索引时，查询耗时约50毫秒——50毫秒不算慢，但我们的大屏每30秒刷新一次，如果同时有几十个并发查询，累积效应就很可观了。加上这个复合索引以后，同样的查询降到约2毫秒——快了25倍。这个提升不只是快的问题，而是让大屏可以放心地高频刷新，不会给数据库造成压力。

## 第21页：05 七重触发器防护盾（章节分隔页）

好，接下来的章节是整个项目最有技术含量、也是我投入最多心血的部分——**七重触发器防护盾。**

传统开发模式是：业务规则写在应用代码里。用户下单 → Python 检查库存 → 扣库存 → 写订单。这个模式在大多数场景下没问题，但存在三个隐患：应用代码可能写错、API 可能被绕过、并发场景下多个请求同时读同时写会产生竞态条件。

我们的思路是反过来的：**把关键业务规则从应用层下沉到数据库引擎层。** 触发器是什么？就是一段由数据库引擎自动执行的程序——在数据写入磁盘之前触发，不满足条件就拒绝写入。即使有人绕过 Flask 直接连 MySQL 写 SQL，这些规则照样生效。

七个触发器，覆盖四个维度的约束：库存安全、骑手类型、骑手状态、寄存点容量。接下来我逐一展示每个触发器的设计逻辑和代码实现。

---

## 第22页：FOR UPDATE 行级锁原理

在讲具体的触发器之前，必须先讲一个核心概念——**FOR UPDATE 行级锁**。它是整个防超卖体系的基石，不理解它就理解不了触发器为什么能工作。

我用一个每个人都能懂的场景来解释。

假设食堂今天最后一份黄焖鸡，库存 stock = 1。你和隔壁寝室的小明同时打开手机下单——两个请求几乎在同一瞬间到达服务器。

如果没有行级锁，会发生什么？线程 A 执行 `SELECT stock FROM dishes WHERE dish_id = 100`，读到了 stock = 1。线程 B 几乎同时执行同样的查询，也读到了 stock = 1。两个线程都判断"还有库存，可以下单"。于是两个订单都成功创建，库存被扣成 -1——一份鸡，卖了两次。这就是经典的**"读-判断-写"竞态条件**，是并发编程中最常见的数据一致性问题。

FOR UPDATE 怎么解决呢？线程 A 执行的 SQL 不是普通的 SELECT，而是：

```sql
SELECT stock FROM dishes WHERE dish_id = 100 FOR UPDATE;
```

多了 `FOR UPDATE` 这四个单词，效果完全不同。线程 A 执行这条语句的时候，数据库引擎在 dish_id = 100 这行数据上**加了一把排他锁**——也叫 X 锁。这把锁的意思是：在我这个事务提交之前，其他任何人不准读、也不准写这行数据。

线程 B 紧接着也执行了 `SELECT ... FOR UPDATE`，但这次它被**阻塞**了——数据库引擎让它等着，因为这一行已经被 A 锁住了。线程 B 只能在等待队列里排队。等线程 A 扣完库存、更新 stock = 0、提交事务、释放锁之后，线程 B 才被唤醒，终于拿到了锁。但此时它读到的 stock 已经是 0 了。0 < 用户的购买数量 1 → 库存不足 → SIGNAL 抛错 → 下单被拒绝。

一份鸡，老老实实只卖了一次。

**FOR UPDATE 的本质，是把"读数据 → 判断条件 → 写结果"这三个独立的步骤，绑定成了一个不可分割的原子操作。** 这是数据库课程里最核心的并发控制概念，也是我们整个防超卖体系的基石。

我们系统里数了一下，FOR UPDATE 一共用在了三个关键位置：触发器1的库存检查、触发器7的容量预检、sp_create_order 的下单预检。每一个位置都是高并发场景下的关键路径。

---

## 第23页：触发器1&2——库存防超卖检查 + 自动扣减

有了 FOR UPDATE 的基础，现在来看触发器的具体设计。

触发器1和触发器2是一对搭档，一个做检查、一个做执行，配合得天衣无缝。

**触发器1（BEFORE INSERT ON order_items）：**

在订单明细行即将写入磁盘之前触发。它做的第一件事就是：`SELECT stock, status FROM dishes WHERE dish_id = NEW.dish_id FOR UPDATE`。注意这里用的是 FOR UPDATE，不是普通的 SELECT——它锁住了这行菜品数据。

然后做两个校验。校验一：这道菜是不是已经下架了？`IF v_status = 0 THEN SIGNAL`——如果下架了，直接抛错，拒绝写入。校验二：库存够不够？`IF v_stock < NEW.quantity THEN SIGNAL`——如果不够，同样抛错。

只有两道校验都通过了，数据才被允许写入 order_items 表。

**触发器2（AFTER INSERT ON order_items）：**

当且仅当触发器1放行、数据成功写入 order_items 表之后，触发器2才自动执行：`UPDATE dishes SET stock = stock - NEW.quantity WHERE dish_id = NEW.dish_id`。库存扣减完成。

我打个比方大家就记住了。触发器1是**守门员**，不符合条件的人直接拦在门外——你连门都进不去。触发器2是**柜台办事员**，合法的人进了门之后他自动帮你办好手续，你不用操心。守门员拦住的人，柜台办事员根本接触不到。正是这种"前置校验 + 后置执行"的分工协作，保证了高并发场景下库存安全万无一失。

如果只靠应用层代码（Python）来做这个校验会怎样？两个服务器实例可能同时读到 stock = 1，同时通过校验，同时扣库存——因为应用层的锁控制不了数据库层的事务隔离。所以这个校验必须写在离数据最近的地方，也就是数据库引擎内部，而且要配合 FOR UPDATE 行级锁。

---

## 第24页：触发器3&4——骑手类型约束

触发器3和4保证了一条简单的铁律：**干线骑手只能做干线的活，楼栋骑手只能做楼栋的活。不能串岗。**

实现方式是：在订单数据写入（INSERT）和更新（UPDATE）两个时间点，都执行同样的校验逻辑：

- 如果 `stage1_rider_id` 指向的骑手，他的 `rider_type` 不是 `Stage1_Trunk` → 直接 SIGNAL 报错："干线骑手类型错误"
- 如果 `stage2_rider_id` 指向的骑手，他的 `rider_type` 不是 `Stage2_Floor` → 同样报错

你可能会问：骑手表不是已经有 ENUM 限死了 rider_type 吗？为什么还要用触发器再校验一遍？这两个保护不在同一个维度。

ENUM 约束的是"骑手自身的类型不能被乱填"——你不能在 riders 表里创建一个类型为 `Stage3_Unknown` 的骑手。触发器约束的是"订单里的骑手分配不能被乱填"——你不能把已经存在的楼栋骑手分配到订单的干线段去。如果一个骑手本身就错了，ENUM 能拦住；但如果骑手本身是对的，只是被错误地分配到了不对的岗位，ENUM 管不着——触发器来管。

为什么要设两个触发器（INSERT + UPDATE）？因为骑手指派可能发生在两个环节——创建订单时同步指派（INSERT），或者先创建订单、后补派骑手（UPDATE）。两个入口都要堵住，一个都不能漏。

---

## 第25页：触发器5&6——骑手状态自动管理

这是我最喜欢的设计——因为它真正体现了"自动化"的价值。**骑手的工作状态，不用任何人手动去改。**

触发器5和6在订单状态变更时自动同步骑手状态。规则非常直观：

- 系统给订单指派了干线骑手（订单进入 Stage1_Assigned） → 这个干线骑手的状态自动变成 Delivering（配送中），表示"他正在跑单，别再给他派新的了"
- 干线骑手把餐送到了寄存柜（订单进入 Arrived_At_Point） → 干线骑手自动释放，状态变回 Idle（空闲），他可以立刻接下一单
- 楼栋骑手取货开始配送（订单进入 Stage2_Assigned） → 楼栋骑手变成 Delivering
- 学生签收（订单进入 Completed） → 楼栋骑手释放为 Idle
- 订单被取消 → 所有关联骑手——不管干线还是楼栋——全部释放

你想想如果没有这个自动机制会怎样？需要有一个调度员盯着屏幕，看到干线骑手送达了，手动在系统里点一下"释放骑手"。如果调度员忙碌、疏忽、或者系统响应慢，骑手就一直是"配送中"状态，永远不会被分配新订单。系统里明明有八个干线骑手，却只显示三个可用——因为另外五个状态都没更新。

有了触发器，这一切自动发生，毫秒级响应。**订单状态是唯一的真相来源**——状态一变，骑手自动跟着变。零延迟，零遗漏，零人工干预。

大屏首页那个"活跃骑手"的 KPI 数字怎么来的？就一句话：`SELECT COUNT(*) FROM riders WHERE status = 'Delivering'`。不需要复杂的聚合查询，直接查骑手表就行了——因为状态是触发器实时维护的，永远准确。

---

## 第26页：触发器7——寄存点容量预检

第七重防护是我后来加上的——加它的原因来自于对原有设计缺陷的反思。

最开始的系统只有 CHECK 约束防爆仓。CHECK 的触发时机是：数据写入的那一刻。问题来了——如果干线骑手已经取了餐、骑到了宿舍楼下、打开寄存柜准备往里放，这时候发现柜子满了——CHECK 约束拒绝写入，整个事务回滚。骑手白跑了一趟。

这个问题叫"事后拦截"——用户和骑手付出了成本之后才被告知不行，体验太差。

触发器7把拦截时机**提前到了下单阶段**。用户点"下单"按钮的那一刻，触发器7执行：`SELECT current_packages, capacity FROM pickup_points WHERE point_id = ? FOR UPDATE`。行级锁锁住这个寄存点 → 检查 `current_packages >= capacity ?` → 满了就直接 SIGNAL 拒绝下单，并告诉用户："该寄存点已满，请选择邻近寄存点下单。"

用户在下单阶段就被引导到有空位的寄存点，他不会产生"白等"的糟糕体验；骑手也不会白跑。

这样一来，系统就有了**两层防护**：第一层在下单时（触发器7 + sp_create_order 里的容量预检），拦截掉99.9%的爆仓风险；第二层在入柜时（chk_capacity CHECK 约束），兜底极端并发场景。两层之间由 FOR UPDATE 保证每层的检查都是原子操作。

---

## 第27页：06 存储过程与事务管理（章节分隔页）

触发器讲完了——它们负责自动校验和自动同步。但如果业务流程涉及多步操作、需要保证"要么全做、要么全不做"，触发器就不够了。这个时候需要存储过程 + 事务管理。

存储过程是什么？就是一段预编译好、存储在数据库内部的程序，你可以通过一个 CALL 语句来调用它。它最大的优势是支持事务——多步操作包在一个事务里，任何一步失败，已经执行的步骤全部回滚。

---

## 第28页：存储过程：sp_create_order（原子下单）

这个存储过程是教科书级的 ACID 事务实现。我把它的逻辑走一遍。

学生点了一份黄焖鸡，点击"下单"。系统调用 `CALL sp_create_order(用户ID, 商家ID, 寄存点ID, 菜品ID, 数量, @订单ID)`。存储过程内部做了五件事：

第一步，获取菜品的当前价格，计算出订单总金额。这个价格是在事务内读取的，保证了价格快照的准确性。

第二步，检查学生的校园卡余额是否足够。不够？直接 SIGNAL 抛错："余额不足"。

第三步，FOR UPDATE 锁定目标寄存点，检查容量。满了？抛错："该寄存点已满，请选择邻近寄存点。"

第四步，扣减学生余额。

第五步，插入订单主记录（orders 表），获取自增主键作为订单ID。

第六步，插入订单明细（order_items 表）。**注意：这步插入会自动触发之前讲的触发器1和触发器2——库存检查和库存扣减。** 如果库存不足，触发器1会在这个位置抛错。

五步操作，全程包在一个 `START TRANSACTION ... COMMIT` 里。开头有一个 `DECLARE EXIT HANDLER FOR SQLEXCEPTION`——如果任何一步出了任何 SQL 异常，这个 Handler 自动执行 ROLLBACK，余额恢复、订单撤销，整个数据库回到这个存储过程执行之前的状态，就像什么都没发生过。

这就是 ACID 中的 A——Atomicity，原子性。要么全成功，要么全撤销，没有"做了一半"的中间态。

---

## 第29页：存储过程：sp_arrive_at_pickup_point（干线送达入柜）

干线骑手到了寄存柜，确认把餐放进去了，系统调用这个存储过程。

逻辑分两步，都在一个事务里完成。第一步：把订单状态推进到 `Arrived_At_Point`，同时记录下 `stage1_completed_at` 时间戳。第二步：寄存点包裹计数加1。

注意第二步更新完包裹计数后，`chk_capacity` CHECK 约束会自动校验。万一此时寄存点恰好满了（前面提到的极端并发场景），CHECK 约束拒绝写入，事务回滚。干线骑手需要把这个包裹送到另一个有空位的寄存点。

事务提交后，触发器6自动检测到订单状态变成了 Arrived_At_Point，它会把干线骑手的状态释放为 Idle——骑手可以立刻去接下一单。

---

## 第30页：存储过程：sp_stage2_deliver（楼栋末端送达）

楼栋骑手把餐送到学生手上后，调用这个存储过程。

第一步：订单状态推进到 Completed，记录 `stage2_completed_at`——最终签收时间。第二步：寄存点包裹计数减1。

第二步有一个看似多余、实则必要的防御性判断：`WHERE current_packages > 0`。在正常的业务流程里，current_packages 不会减到负数——因为有 CHECK 约束兜底。但加上这个 `WHERE` 条件，可以在极端并发场景下避免 CHECK 抛错导致整个事务不必要的回滚。多一层保护，永远不嫌多。

事务提交后，触发器6检测到 Completed 状态，自动释放楼栋骑手为 Idle。同时，这条订单的三个时间戳全部填写完毕——created_at（下单）、stage1_completed_at（入柜）、stage2_completed_at（签收）——完整的物流时间轴可供审计。

---

## 第31页：存储过程：sp_cancel_order（游标退款 + 库存恢复）★

这是四个存储过程中最有技术含量的一个，因为它用到了 MySQL 的一个高级特性——**游标**。

取消订单不是简单地把 order_status 改成 Cancelled 就完了。这个存储过程需要做四件事：

第一，**退款。** `UPDATE users SET balance = balance + 订单金额 WHERE user_id = 下单用户`。用户的余额加回之前扣掉的金额。

第二，**恢复库存。** 这是最难的一步。一个订单不是只点了一道菜——你可能同时点了黄焖鸡、柠檬水、羊肉串，每道菜的数量还不同。怎么把每道菜的库存逐一加回去？**用游标。**

游标的工作原理类似于：你在 `order_items` 表里打开一个"指针"，它指向这个订单的第一条明细（黄焖鸡，2份）。你用 FETCH 语句把这条记录的 dish_id 和 quantity 读出来，执行 `UPDATE dishes SET stock = stock + 2 WHERE dish_id = 黄焖鸡ID`。然后游标自动指向下一条明细（柠檬水，1份）……如此循环，直到所有明细遍历完毕，关闭游标。

这就是 MySQL 中处理"一对多"关系的标准模式——用游标逐行遍历子表，对每一行执行定制化的操作。

第三，**释放寄存点容量。** 如果这个订单已经入柜了（状态是 Arrived_At_Point 或 Stage2_Assigned），寄存点包裹计数减1。

第四，**标记取消。** 把订单状态改为 Cancelled。这个更新会触发触发器6——自动释放订单关联的所有骑手。

四步操作，一个事务，任何一步失败全部回滚。特别要注意：存储过程开头有一个 IF 判断——如果订单已经是 Completed 或 Cancelled 状态，直接抛错拒绝。这是防止重复退款的安全锁。

## 第33页：寄存点饱和度预警视图

`vw_pickup_point_analytics` 是我们大屏的"数据引擎"。

它的逻辑很简单：对每个寄存点，用 `current_packages / capacity * 100` 算出饱和度百分比，再 LEFT JOIN 一个子查询统计各寄存点当前在途订单数。

## 第34页：商家销售排行视图（窗口函数）

`vw_merchant_sales_rank` 用到了 MySQL 8.0 才引入的窗口函数。

传统的排名查询要怎么写？子查询套子查询，再配合用户变量漂移——SQL 又长又难维护，性能还差。

有了窗口函数，一行搞定：

```sql
RANK() OVER (ORDER BY SUM(total_amount) DESC) AS sales_rank
```

数据库引擎自己会在内存里对结果集做排序和排名，不需要你写复杂的自连接。

`RANK()` 的特点是同分同名次——如果蜜雪冰城和川湘木桶饭这个月都卖了5000块，它们并列第一，下一个商家是第三名（1, 1, 3）。如果你的业务需要连续名次（1, 1, 2），换 `DENSE_RANK()` 就行。选择哪个函数取决于业务语义——我们这个场景用 RANK() 更合理，因为"并列第一"这个词本身就是有价值的业务信息。

## 第35页：AI Text-to-SQL 自然语言查询

这个功能是我们为了降低系统使用门槛而加上的。不是所有运营人员都会写 SQL——但是他们都会说话。

我演示一个真实例子。管理员打开大屏的 AI 查询入口，输入："今天哪个寄存点快满了？"系统拿到了这个自然语言问题，从 `app.py` 里的 Schema 上下文获取完整的表结构信息——七张表、所有字段、外键关系——然后把"问题 + Schema"一起发给 DeepSeek。

DeepSeek 理解了问题意图和表结构之后，自动生成了这条 SQL：

```sql
SELECT point_name, saturation_pct
FROM vw_pickup_point_analytics
ORDER BY saturation_pct DESC LIMIT 1;
```

系统执行这条 SQL，把结果展示给管理员："3期智能寄存柜，饱和度100%，已爆仓。"全程管理员没有写一行代码。

当然，安全机制必不可少。AI 可能产生幻觉，但它生成的 SQL 在我们的系统里会被严格审查：必须是以 SELECT 开头的只读查询；如果 SQL 里包含了 INSERT、UPDATE、DELETE、DROP、ALTER 这些危险关键词——直接拦截，不执行。AI 只负责"查"，绝不负责"改"。数据库数据不会被破坏。

每次调用成本不到一分钱——DeepSeek 的 API 定价是国产大模型里最有竞争力的。

## 第42页：七大创新点

我把整个项目的创新归纳为七个点。这七个点不是随便罗列的——每一项都对应一个真实的技术挑战和我们的解决方案。

**第一，FOR UPDATE 行级锁防超卖。** 高并发场景下，"读-判断-写"不再是三步分散的操作，而是一个不可分割的原子操作。这是数据库课程核心考点之一，我们不是只在理论上学过，而是在项目里真正落地了。

**第二，七重触发器防护体系。** 库存方向两道（检查+扣减），骑手类型方向两道（INSERT+UPDATE），骑手状态方向两道（INSERT+UPDATE），容量预检方向一道。四个维度，七个触发器，构成了数据库层的自动化安全防线。约束写在离数据最近的地方。

**第三，两段式六态订单状态机。** ENUM 限死合法状态 + 六态流转 + 双骑手追踪 + 三时间戳审计，订单全生命周期标准化、可追溯。

**第五，RANK() OVER 窗口函数实现商家排名。** MySQL 8.0 新特性的实战应用——一行SQL替代复杂的子查询和变量漂移，高效、简洁、可维护。

## 附：常见答辩问题预案

### Q1：为什么不用 Redis 做库存扣减？

Redis 确实比 MySQL 快——内存操作，单机十万QPS很轻松。但它和 MySQL 是两套独立的存储系统。Redis 扣了库存，但 MySQL 写订单失败了怎么办？你需要引入分布式事务（两阶段提交、TCC、Saga），复杂度直接上了几个数量级。

我们的场景是什么？一个校园、一万多学生、午餐高峰几百单——这个量级 MySQL 的 FOR UPDATE 行级锁完全能应付。每秒几百次行锁操作，InnoDB 轻松处理。选技术不能光看"快不快"，要看"够不够"——在我们的场景下，MySQL 一致性更可靠、架构更简单。

### Q2：七个触发器会不会拖慢写入性能？

每个触发器只做轻量级的操作——查一行数据、更新一行数据、或做简单判断。所有操作都在内存中的缓冲池完成，不涉及磁盘I/O。开销在微秒到毫秒级别。

而且触发器的逻辑写在数据库引擎内部——它在执行你的 UPDATE 语句的同时"顺便"就把校验做掉了，没有额外的网络往返。如果你把这些校验逻辑写在 Python 应用层，你需要："SELECT 查询数据 → 网络传输到Python → Python判断 → 网络传回MySQL → UPDATE"——两次网络往返。触发器把这一切省掉了，都在引擎内部完成。实际上可能比你写应用层校验更快。

### Q3：如果寄存点真的爆仓了怎么办？

两层防护机制。第一层：用户下单时，sp_create_order 和触发器7 双重容量预检——满了直接拒绝下单，引导用户选邻近寄存点。这一层覆盖了99.9%的场景。

第二层：在极端并发场景下——比如两个用户恰好同时下单、恰好抢占最后一个格子、第一层恰好都放过了——chk_capacity CHECK 约束会在入柜操作时兜底拦截。当然，这会让干线骑手白跑一趟，但这种情况的概率极低，我们接受这个极小概率的代价。

运营层面，大屏实时展示各寄存点饱和度——超过80%黄色预警，运营人员提前调度楼栋骑手加速清空积压包裹。技术手段和运营手段结合。

### Q4：AI 生成的 SQL 会不会有安全风险？

三层防护。第一层：提示词里明确要求"只生成 SELECT 语句"。第二层：Python 层检查生成的 SQL 必须以 SELECT 开头。第三层：扫描 SQL 中是否包含危险关键词——INSERT、UPDATE、DELETE、DROP、ALTER、TRUNCATE、CREATE、EXEC——任何一个出现都直接拦截。AI 可能"幻觉"，但金字塔底的那层 Python 校验是100%确定的。

### Q5：你们和美团/饿了么的校园配送有什么不同？

他们的模式是：骑手从校外商家取餐，送到校门口。我们在这个基础上加了第二段——从校门口到寝室门口。美团优化的是校门口的交接效率（智能取餐柜、分时段取餐），我们解决的是进门以后的末端分发。

技术层面最大的不同：我们强调数据库层面的约束保护。我们的系统即使应用层代码全部扔掉重写，数据库的结构性约束——CHECK、FOREIGN KEY、触发器——仍然能保证数据不出错。这是数据库课程设计的核心训练。

### Q6：你这七个触发器会不会有循环触发的问题？

不会。因为每个触发器监听的事件和它执行的操作不在同一个循环链上。比如触发器1监听的是 `BEFORE INSERT ON order_items`，它执行的操作是 SIGNAL 抛错（不会触发其他触发器）。触发器2监听的是 `AFTER INSERT ON order_items`，它执行的是 `UPDATE dishes`——但 dishes 表上没有任何 AFTER UPDATE 触发器去更新 order_items 表。没有形成闭环，就不会循环触发。MySQL 也有一个安全机制——触发器嵌套深度超过一定层数会自动报错终止，防止死循环。
